My Approach

Just a couple of notes on how I approached the project. Not all of this stuff was called for, perhaps, but I wanted to demonstrate where I'm coming from as a developer.

Full-Stack

As this is a full-stack position, I've built my project using Django. That's probably ridiculous overkill -- I'm barely using any models, and there's no database to speak of -- but I wanted to show my work. And to make the Tachyon markup easier to manage, I wanted to lay it out with a loop (without relying on Javascript). So I'm importing your JS file as JSON using Python, and passing it from the view to the template, where it's laid out as server-side HTML.

Tachyons

This was a new framework for me. I've been a little uncomfortable, in the past, with styling conventions that lean so heavily on class names, but once I got the hang of it, I definitely appreciated the ease and speed with which I could lay everything out. I ended up writing much less CSS than usual. (The only area where I felt constrained was in progressive enhancement -- styling certain components based on the presence of Javascript. See Progressive Enhancement, below.)

Mobile-Responsiveness

All of my front-end work is mobile-first, and I took that approach here, too. Obviously I'm still getting the hang of Tachyons and its breakpoints, but the project should still look pretty good on a phone.

Progressive Enhancement

I believe the web is for everybody, so I've always developed with progressive enhancement in mind. To that end, I try to make my web pages look and feel as snazzy and complete as possible with just bedrock HTML and CSS, and then layer Javascript on for extra effect. In this case, users without Javascript enabled will still get all of the images and descriptions of all the reporters and photographers -- just in a simple list form, with an introductory navigation menu. More advanced browsers, meanwhile, will see that nav menu transformed into the full carousel treatment.

Accessibility

To make sure that all users could make sense of the page, I started with a clean, valid document outline and helpful alt text.

Because the carousel format could be confusing for someone consuming the site with a screen reader, the carousel itself is built out of a nav menu. That way, users should be able to navigate the list of journalists using a keyboard, opening their names and descriptions in turn.

It's not in the design, but I've also included a start/stop button for the carousel animation, as users with motion issues could find the endless scroll disorienting. (Even I found myself wanting to get off the ride now and then!)

There are shortcomings to my approach, though. For instance, to pull off the infinite scroll, I had to clone menu items and append them to the end of the list. A blind user who didn't know they were navigating through a carousel would be confused to find each staffer coming up again and again and again...

i18n

It's a minor thing, but using Django also allowed me to easily make some of the more static "theme" strings translatable. Even the start/stop button was made translatable, by passing the translated strings to the element as data attributes, and making the Javascript file do its text replacement using those values.
